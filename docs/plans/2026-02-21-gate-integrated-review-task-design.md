# Gate Integrated Review Task Design

## 배경

현재 `.roadmap/ai_agent_roadmap.md`는 게이트별 DEV 태스크가 상세하게 정의되어 있으나,
게이트 단위 완료 판단(통합 리뷰)을 독립 태스크로 고정한 구조는 없다.
이 때문에 "개별 DEV 완료"와 "게이트 목표 달성" 사이의 검증 책임이 분리되지 않는 문제가 있다.

## 의사결정

- 각 게이트(G1~G6)마다 통합 리뷰 DEV 태스크 1개를 추가한다.
- 통합 리뷰 DEV는 해당 게이트의 모든 DEV를 `depends_on`으로 갖는다.
- 통합 리뷰 DEV의 `approval`은 모두 `manual`로 고정한다.
- `verify`는 게이트 위험도에 맞춰 설정한다.
  - G1: L1
  - G2: L2
  - G3: L2
  - G4: L2
  - G5: L3
  - G6: L3

## 접근안 비교

### A. 게이트당 DEV 1개 추가 (채택)

- 장점: 의존성/완료 기준/승인 경계가 게이트 단위로 명확해진다.
- 단점: 태스크 수가 6개 증가한다.

### B. 게이트별 리뷰 Epic 추가

- 장점: 구조 구분이 선명하다.
- 단점: 문서 깊이가 늘고 탐색 비용이 커진다.

### C. 기존 마지막 DEV DoD에 흡수

- 장점: 태스크 수를 늘리지 않는다.
- 단점: 게이트 종료 책임이 기존 구현 태스크에 섞여 추적성이 낮아진다.

## 통합 리뷰 DEV 템플릿

각 게이트 통합 리뷰 DEV는 아래 공통 DoD를 따른다.

1. 해당 게이트 범위 DEV DoD 충족 여부가 체크리스트로 정리되어 있다.
2. 게이트 선언문 달성 여부와 근거(테스트/스모크/문서)가 기록되어 있다.
3. 문서 영향(`.specs/`, `docs/adr/`, runbook 등)과 불일치 여부가 점검되어 있다.
4. 미해결 리스크와 후속 액션이 명시되어 있다.
5. 기존 CI가 모두 통과한다.

## 번호 정책

- 기존 DEV-001~DEV-047 뒤에 DEV-048~DEV-053을 순차 추가한다.
- Epic 번호는 기존 EPIC-34 뒤에 EPIC-35~EPIC-40을 순차 추가한다.

## 기대 효과

- 게이트 단위 Go/No-Go 기준이 태스크로 명시되어 리뷰 책임이 선명해진다.
- Sudocode 이슈/PR로 연결할 때 "게이트 종료" 산출물이 일관되게 남는다.
- 추후 운영 리포트에서 게이트별 완료 품질을 비교하기 쉬워진다.
